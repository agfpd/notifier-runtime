# 02 — Быстрый старт

**Русский** · [English](../02-quickstart.md)

Здесь вы пройдёте путь от установленного хоста iapeer до живого триггера, который срабатывает. Нужен уже настроенный iapeer (если нет — сначала запустите `iapeer onboard`).

## 1. Установите рантайм

Установите пакет и дайте iapeer провижнить его пиров:

```sh
npx -y @agfpd/notifier-runtime    # self-install: bin на PATH + манифест рантайма
iapeer install-runtime notifier   # iapeer читает манифест, провижнит timer + watcher
```

Первая команда кладёт самодостаточный лаунчер на ваш `PATH` и пишет манифест рантайма. Вторая передаёт манифест в iapeer, который провижнит пиров `timer` и `watcher`, пишет их launchd-джобы и запускает их. Теперь в реестре два новых пира.

## 2. Зарегистрируйте триггер

Регистрация — это вызов инструмента, который делает агент-пир: `send_to_peer` с JSON-телом. Человек в чате шлёт только текст; структурный вызов вроде этого — то, что агент делает от своего имени. С пира:

```
send_to_peer(timer, {"when": "@every 1m", "message": "тик", "target": "self"})
```

Пир получает ответ с подтверждением триггера и списком своих активных. `target: "self"` направляет сигнал обратно пиру, который зарегистрировал триггер — так что через минуту этот пир получит сообщение `тик` в свежей сессии.

Запланируйте сообщение раз в день:

```
send_to_peer(timer, {"when": "0 9 * * *", "message": "Пора на дейли", "target": "self"})
```

Или заставьте `watcher` форвардить совпадение в логе:

```
send_to_peer(watcher, {"script": "tail -F /var/log/app.log | grep ERROR", "target": "self"})
```

Каждая строка, которую скрипт печатает в stdout, приходит на `self` отдельным сигналом.

## 3. Проверьте, что зарегистрировано

Спросите у пира его триггеры в любой момент:

```
send_to_peer(timer, {"cmd": "list"})
```

Или проверьте всю проекцию из shell, по обоим пирам:

```sh
notifier-runtime doctor          # для человека
notifier-runtime doctor --json   # машиночитаемо
```

`doctor` показывает каждый триггер, парсится ли он и — для таймеров — время следующего срабатывания.

## 4. Удалите триггер

```
send_to_peer(timer, {"cmd": "unregister", "id": "<id>"})
```

`id` — тот, что в ответе при регистрации (или `id`, который вы выбрали). Повторная регистрация с тем же `id` *заменяет* триггер; чтобы отредактировать, снимать его заранее не нужно.

## Что дальше

- Сообщение попадает в **новую сессию** — пишите его самодостаточным, без отсылок к контексту, которого у пира-получателя не будет.
- Триггер принадлежит пиру, который его зарегистрировал. Пир может перечислять, править и удалять только свои. См. [05 — Регистрация триггеров](05-регистрация-триггеров.md).
- Расписание настраивается в [03 — Timer](03-timer.md); watcher — в [04 — Watcher](04-watcher.md).
