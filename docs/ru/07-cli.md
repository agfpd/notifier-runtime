# 07 — CLI

**Русский** · [English](../07-cli.md)

У команды `notifier-runtime` четыре подкоманды. В обычной работе вы не запускаете ни одну вручную — iapeer сам вызывает `self-install`, `self-config` и `run`, а `doctor` — та, к которой потянетесь вы. Они описаны здесь, чтобы вы знали, что рантайм делает за вас.

```
notifier-runtime                 # self-install (контракт npx)
notifier-runtime self-install    # явный self-install (идемпотентно)
notifier-runtime run             # всегда-включённый run-loop (точка входа лаунчера)
notifier-runtime self-config     # пер-пировый хук конфигурации (вызывается фундаментом)
notifier-runtime doctor [--json]
```

## `self-install`

Кладёт самодостаточный лаунчер на ваш `PATH` и пишет манифест рантайма. Голая команда без аргументов делает то же — это контракт `npx -y @agfpd/notifier-runtime`, по которому iapeer вызывает пакет без аргументов.

Бинарь идёт в `IAPEER_BIN_DIR` или в `~/.local/bin`, если та не задана. Это скомпилированный снимок (`bun build --compile`), поэтому он работает без отдельного рантайма на `PATH`. Манифест пишется в `<IAPEER_ROOT>/runtimes/notifier/runtime.json` *после* того, как бинарь на месте. Вся операция идемпотентна — запустите снова, и получите байт-в-байт идентичный манифест. Что в манифесте — см. [09 — Архитектура](09-архитектура.md).

## `run`

Всегда-включённый run-loop — точка входа, которую лаунчер iapeer запускает под launchd. Он диспатчит по разрешённой личности: `timer` запускает планировщик (TIME), `watcher` — супервизор (EVENT). Цикл грузит триггеры пира, заново вычерпывает сигналы, застрявшие в журнале от предыдущего процесса, слушает сообщения-регистрации на stdin и запускает триггеры по мере наступления срока. Завершается чисто по SIGINT/SIGTERM.

Вы не запускаете это напрямую; iapeer поднимает его как `notifier-timer` и `notifier-watcher`.

## `self-config`

Пер-пировый хук, который фундамент вызывает при создании или онбординге notifier-пира. Он пишет регистрационную самодокументацию роли в описание локального профиля пира, сохраняя идентичность пира (`intelligence: absent`). Какую роль он настраивает, хук читает из `IAPEER_PEER_PERSONALITY`. Код выхода `0` означает «настроено».

## `doctor`

Та команда, что вы запустите сами. Она валидирует всю проекцию триггеров — **обе** роли, независимо от того, какого пира вы спрашиваете — так что проверить всё можно из одного места.

```sh
notifier-runtime doctor
```

```
notifier-runtime doctor (identity notifier-timer)
triggers: 3 (valid 3, invalid 0)
  [ok] (time) alice -> self    0 9 * * *    next=2026-06-18T09:00:00.000Z
  [ok] (time) ops -> oncall    @every 30m   next=2026-06-17T18:30:00.000Z
  [ok] (event) ops -> ops      monitor-disk    script-ok  heartbeat=60s
```

Для каждого триггера выводится роль, владелец, получатель, расписание или скрипт, валиден ли он и — для таймеров — время следующего срабатывания. Невалидный триггер показывает причину. Добавьте `--json` для машиночитаемого вывода (те же данные плюс сводный счётчик), полезного для внешнего верификатора.

```sh
notifier-runtime doctor --json
```
