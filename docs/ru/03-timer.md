# 03 — Timer

**Русский** · [English](../03-timer.md)

Пир `timer` шлёт сообщение по расписанию. Агент-пир регистрирует его вызовом `send_to_peer` с `when` (расписание), `message` и `target`. По желанию добавляет `check`, чтобы закрыть сообщение условием.

```
send_to_peer(timer, {"when": "0 9 * * *", "message": "Дейли", "target": "self"})
```

## Расписание: `when`

`when` — это либо cron-выражение из пяти полей, либо интервал `@every`.

### Cron

Пять полей через пробел, в локальном системном времени, с гранулярностью в минуту:

```
┌─ минута            0–59
│ ┌─ час             0–23
│ │ ┌─ день месяца   1–31
│ │ │ ┌─ месяц       1–12
│ │ │ │ ┌─ день недели 0–7 (0 и 7 — оба воскресенье)
│ │ │ │ │
0 9 * * *
```

Каждое поле принимает:

| Форма | Значение | Пример |
|-------|----------|--------|
| `*` | каждое значение | `* * * * *` — каждую минуту |
| `a` | одно значение | `0 9 * * *` — в 09:00 ежедневно |
| `a-b` | диапазон | `0 9-17 * * *` — в начале каждого часа с 09:00 до 17:00 |
| `*/n` | каждое n по полю | `*/15 * * * *` — каждые 15 минут |
| `a-b/n` | каждое n внутри диапазона | `0 9-17/2 * * *` — 09:00, 11:00, … 17:00 |
| `a,b,c` | список | `0 9,13,18 * * *` — в 09:00, 13:00, 18:00 |

День месяца и день недели подчиняются стандартному правилу OR из Vixie-cron: когда **оба** поля ограничены (ни одно не `*`), день совпадает, если совпало **любое**. `0 9 13 * 5` срабатывает в 09:00 тринадцатого числа *и* каждую пятницу — не только по пятницам, выпавшим на 13-е.

Расписание идёт по локальному времени хоста. Специальной обработки перехода на летнее время нет; на хосте без перехода (например UTC+3) это не имеет значения.

### Интервалы

`@every <длительность>` срабатывает с фиксированным периодом:

```
@every 30m      каждые 30 минут
@every 1h30m    каждые 90 минут (единицы складываются)
@every 45s      каждые 45 секунд
@every 2d       каждые 2 дня
```

Единицы — `s`, `m`, `h`, `d`, и при сочетании они суммируются. Минимальный интервал — одна секунда. Интервал привязывается к моменту старта триггера (или переармируется при правке) и отсчитывается вперёд от него — не от начала часа или суток.

### Пропущенные срабатывания сворачиваются

Если рантайм был выключен в одно или несколько расписанных времён — перезагрузка хоста, ноутбук в спящем режиме — он **не** проигрывает их при старте. Каждое расписание перескакивает к ближайшему будущему срабатыванию. Пять пропущенных дейли становятся одним предстоящим, а не пятью разом.

## Гейт: `check`

`check` — это команда, запускаемая прямо перед тем, как сообщение было бы отправлено. Сообщение уходит, только если проверка завершилась с кодом `0`; любой ненулевой код тихо пропускает это срабатывание. Соглашение — как в shell: `0` означает «условие выполнено, слать», как `test` или `grep -q`.

```
send_to_peer(timer, {
  "when": "*/10 * * * *",
  "check": "/usr/local/bin/queue-nonempty",
  "message": "В очереди работ есть ожидающие задачи",
  "target": "ops"
})
```

Проверка работает по принципу fail-safe: если команду не нашли, она не исполняема, истекла по таймауту или убита сигналом — срабатывание **пропускается**, сообщение не отправляется, а сбой громко логируется. Сломанный гейт никогда не даёт ложной тревоги. У проверки есть таймаут 30 секунд, чтобы зависший гейт не заклинил расписание.

## Поля

| Поле | Обязательно | Значение |
|------|-------------|----------|
| `when` | да | cron-выражение или интервал `@every` |
| `message` | да | текст, доставляемый получателю (`msg` принимается как алиас) |
| `target` | да | пир-получатель или `"self"` для пира-отправителя |
| `check` | нет | команда; слать, только если код выхода `0` |
| `id` | нет | стабильное имя триггера; опустите — выведется из содержимого |
| `topic` | нет | короткая метка, которую несёт сигнал (≤200 символов) |
| `fallback` | нет | пир(ы) эскалации, если получатель недостижим — см. [06 — Эскалация](06-эскалация.md) |

`message` доставляется в **новую сессию** у получателя. Пишите его самодостаточным — у пира-получателя не будет того контекста, что был у вас при регистрации триггера.

## Правка и удаление

Зарегистрируйте заново с тем же `id`, чтобы заменить триггер на месте. Таймер с неизменным `when` сохраняет своё место в расписании; смените `when` — и он переармируется от текущего момента. Список и удаление:

```
send_to_peer(timer, {"cmd": "list"})
send_to_peer(timer, {"cmd": "unregister", "id": "daily-standup"})
```

Полный формат сообщения и правила владения — в [05 — Регистрация триггеров](05-регистрация-триггеров.md).
